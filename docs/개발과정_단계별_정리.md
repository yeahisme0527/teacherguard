## Step 1. 기획 & 앱 설계

### 배경

학부모의 무리한 요구, 욕설, 협박성 메시지로 인한 교권침해 사례가 증가함.  
교사들이 직접 이를 필터링하면서 감정 소모가 크고, 증거 수집도 어려움.

### 해결 방향

> **"메시지가 교사에게 닿기 전에 AI가 먼저 검사한다"**

- 학부모가 메시지를 쓸 때 실시간으로 경고 표시
- 심각한 메시지는 아예 교사에게 전달되지 않게 차단
- 부적절 메시지를 자동으로 보관해 신고서 생성에 활용

### 설계한 기능 목록

```
✅ 메시지 키워드 탐지 (욕설·협박·모욕 등 8가지 유형)
✅ 전송 전 실시간 경고 배너
✅ 부적절 메시지 자동 차단 & 블러 처리
✅ AI 자동 응답
✅ 교사 대시보드 (모니터링)
✅ 보관함 (증거 관리)
✅ 교권침해 신고서 자동 생성
✅ 운영시간 외 자동 안내
```

### 기술 선택 이유

| 기술 | 선택 이유 |
|------|---------|
| **Streamlit** | Python만으로 웹앱 완성 가능. 프론트엔드 별도 개발 불필요 |
| **Supabase** | 무료 플랜으로 PostgreSQL + 인증 한번에 해결 |
| **HuggingFace** | 무료 Inference API로 모델 서버 없이 AI 기능 구현 |
| **GitHub + Streamlit Cloud** | 코드 push만 하면 자동 배포. 무료 |

## Step 3. 메시지 분석 엔진

### 만든 것

- `ABUSE_TYPES` 딕셔너리 — 8가지 갑질 유형 정의
- `VARIANT_PATTERNS` 딕셔너리 — 우회 표현 탐지 패턴
- `analyze_message()` 함수 — 핵심 분석 로직
- `blur_text()` 함수 — 욕설 ⬛ 처리
- `get_auto_response()` 함수 — 상황별 자동 응답

### ABUSE_TYPES 구조 설계

```python
ABUSE_TYPES = {
    "욕설/폭언": {
        "keywords": ["병신", "미친", "개새", "시발", ...],
        "severity": 4,   # 심각도 1~5
        "emoji": "🤬",
        "color": "#D32F2F",
    },
    "협박/위협": { "severity": 5, ... },
    "모욕/비하": { "severity": 3, ... },
    ...
}
```

**8가지 유형 선정 기준**: 교권침해 실제 사례를 분석해 가장 자주 등장하는 패턴으로 구성.

### 변형어 문제 발견 → VARIANT_PATTERNS 추가

초기 키워드 탐지만으로는 `ㅂㅅ`, `병.신`, `멍처이` 같은 우회 표현을 못 잡는 문제 발생.

```python
VARIANT_PATTERNS = {
    "병신": ["ㅂㅅ", "뵹신", "뼝신", "병.신", "병 신"],
    "시발": ["ㅅㅂ", "씨발", "십8", "시팔"],
    "멍청이": ["멍처이", "멍청ㅇ", "멍치이"],  # 나중에 추가
    ...
}
```

### analyze_message() 처리 순서

```
① 빈 문자열 체크
② ABUSE_TYPES 키워드 순회 → detected_types 수집
③ VARIANT_PATTERNS 변형어 순회 → 욕설/폭언에 포함
④ 강도 보정어 체크 ("진짜", "정말" → severity +1)
⑤ (선택) AI 모델 결과 병합
⑥ severity → emotion_level 변환 → 결과 dict 반환
```

## Step 7. Supabase DB 설계

### 배경

Streamlit의 `session_state`는 새로고침하면 초기화됨.  
채팅 기록·보관함·차단 목록을 영구 저장할 DB 필요.

### 선택: Supabase (PostgreSQL)

- 무료 플랜: 500MB 저장, 2개 프로젝트
- 인증(Auth) + DB를 하나로 해결
- Python 클라이언트(`supabase-py`) 제공
- Row Level Security(RLS)로 데이터 접근 제어

### 테이블 설계

```sql
-- 1. 사용자 프로필
CREATE TABLE profiles (
    id            uuid PRIMARY KEY,   -- auth.users 와 연결
    email         text,
    display_name  text,
    school_region text,               -- 나중에 추가
    school_type   text,
    school_name   text,
    invite_code   text UNIQUE         -- 나중에 추가
);

-- 2. 채팅 메시지
CREATE TABLE messages (
    id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         uuid,             -- 발신자
    room_id         uuid,             -- 교사의 user_id (나중에 추가)
    sender_role     text,             -- 'parent' / 'ai' / 'teacher'
    text            text,
    blurred_text    text,
    blocked         boolean,
    is_profanity    boolean,
    emotion_level   int,
    severity        int,
    types_detected  jsonb,
    detected_keywords jsonb,
    display_time    text
);

-- 3. 차단 목록
CREATE TABLE blocked_users (
    teacher_id      uuid,
    blocked_user_id text,
    UNIQUE (teacher_id, blocked_user_id)
);
```

### 트리거 설정 (자동 프로필 생성)

회원가입 시 `auth.users`에 행이 생기면 자동으로 `profiles` 에도 생성:

```sql
CREATE FUNCTION handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO profiles (id, email, display_name)
    VALUES (NEW.id, NEW.email, NEW.raw_user_meta_data->>'display_name');
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER on_auth_user_created
    AFTER INSERT ON auth.users
    FOR EACH ROW EXECUTE FUNCTION handle_new_user();
```

## Step 8. DB 모듈 (db.py)

### 만든 것

`db.py` — Supabase와의 모든 통신을 담당하는 전용 모듈.

### 핵심 설계 원칙

**① 모든 함수는 `(bool, data)` 튜플 반환**

```python
# 성공 시
return True, {"user_id": "...", "email": "..."}
# 실패 시
return False, "오류 메시지"
```

→ 호출 측에서 항상 `ok, data = db.함수()` 로 성공 여부를 먼저 체크.

**② 모든 함수에 try/except → 앱이 절대 죽지 않음**

```python
def fetch_messages():
    try:
        resp = client.table("messages")...execute()
        return True, resp.data
    except Exception:
        return False, []   # DB 오류가 나도 빈 리스트 반환
```

**③ `@st.cache_resource` 로 클라이언트 1회 생성**

```python
@st.cache_resource(show_spinner=False)
def get_client():
    return create_client(url, key)
```

앱 실행 중 DB 클라이언트를 한 번만 만들어 재사용.

## Step 11. KST 시간 버그 수정

### 발견한 버그

Streamlit Cloud는 **UTC 서버**에서 실행됨.  
오후 2시(한국)에 앱을 열면 `datetime.now()`가 오전 5시를 반환 → **"운영시간 외"** 로 잘못 판정.

### 수정 방법

**Before:**
```python
now = datetime.now()   # UTC 기준 → 오전 5시
```

**After:**
```python
KST = timezone(timedelta(hours=9))

def now_kst() -> datetime:
    return datetime.now(KST)   # KST 기준 → 오후 2시

now = now_kst()   # 모든 시간 처리를 이 함수로 통일
```

코드 전체에서 `datetime.now()` 를 전부 `now_kst()` 로 교체.

## Step 12. AI 혐오표현 탐지 연동

### 배경

키워드 사전에 없는 신조어·변형어는 탐지 불가. AI 모델 보강 필요.

### 선택한 모델: `smilegate-ai/kor_unsmile`

- 한국어 혐오표현 멀티라벨 분류 모델
- HuggingFace Inference API로 서버 없이 호출 가능
- 무료 (토큰 발급 필요)

### hate_model.py 구현

```python
API_URLS = [
    "https://router.huggingface.co/hf-inference/models/smilegate-ai/kor_unsmile",
    "https://api-inference.huggingface.co/models/smilegate-ai/kor_unsmile",
]

def classify(text, threshold=0.25):
    for url in API_URLS:           # 두 엔드포인트 순서대로 시도
        resp = requests.post(url, json={"inputs": text}, timeout=25)
        if resp.status_code == 200:
            return 결과_dict
    return None   # 둘 다 실패 → 키워드 탐지만 사용 (앱은 정상 동작)
```

### API 엔드포인트 이중화

처음에는 구형 엔드포인트만 사용했다가 실패하는 경우 발생 → 신형 router 엔드포인트를 1순위로, 구형을 폴백으로 설정.

### 적용 시점 결정

### 사용 API: NEIS Open API (교육부 나이스)

- 전국 유치원·초등·중·고·특수학교 목록 제공
- 무료, 즉시 발급
- 엔드포인트: `https://open.neis.go.kr/hub/schoolInfo`

### schools.py 구현 핵심

```python
@st.cache_data(ttl=86400)        # 1일 캐시 (API 과호출 방지)
def _fetch_from_neis(region_code, school_type):
    names = []
    for page in range(1, 6):     # 페이지당 1000개, 최대 5000개
        r = requests.get(NEIS_API_URL, params={...})

        names.extend(row["SCHUL_NM"] for row in rows)
        if len(rows) < 1000:
            break                # 마지막 페이지에서 멈춤
    return sorted(set(names))
```

## Step 15. 초대코드 & 방 연결 시스템

### 배경

학교가 다른 여러 교사가 앱을 쓰면, 학부모 메시지가 전체 교사에게 노출되는 문제.  
교사별 완전 분리된 채팅방 필요.

### 설계 결정: 초대코드 방식

| 방식 | 장점 | 단점 |
|------|------|------|
| 초대코드 | 간단, 직관적 | 코드 유출 시 외부인 접근 가능 |
| 교사 목록에서 선택 | 안전 | 교사 가입 필요, 복잡 |
| QR코드 | 편리 | 구현 복잡 |

→ **초대코드** 선택. 6자리로 단순하고 알림장에 적기 쉬움.

### room_id 개념 도입

```
기존: messages.user_id = 발신자의 ID
새로: messages.room_id = 교사의 user_id (방 식별자)
```

학부모가 전송한 메시지의 `room_id` = 교사 ID  
교사가 `WHERE room_id = 본인 ID` 로 자신의 방 메시지만 조회

### DB 마이그레이션

```sql
ALTER TABLE profiles ADD COLUMN IF NOT EXISTS invite_code text UNIQUE;
ALTER TABLE messages  ADD COLUMN IF NOT EXISTS room_id uuid;
CREATE INDEX messages_room_id_idx ON messages(room_id);
```

### 초대코드 생성 로직

```python
chars = "ABCDEFGHJKLMNPQRSTUVWXYZ23456789"  # 혼동 글자(0,1,I,O) 제외
code = "".join(secrets.choice(chars) for _ in range(6))
```

`secrets` 모듈 사용 → 보안적으로 안전한 난수 생성.

## 🔑 개발하면서 배운 핵심 교훈

### 1. Streamlit의 재실행 모델 이해가 필수

모든 상호작용마다 전체 스크립트가 재실행됨.  
`session_state` 없이 변수를 쓰면 버튼 클릭할 때마다 초기화됨.

### 2. 폴백(Fallback) 우선 설계

DB 없어도 작동, AI 없어도 작동, NEIS 키 없어도 작동.  
모든 기능이 없을 때의 동작 방식을 먼저 설계.

### 3. UTC vs KST — 서버 환경의 시간대 함정

로컬(PC)에서 개발하면 `datetime.now()`가 KST.  
서버에 배포하면 UTC. 반드시 명시적 timezone 처리 필요.

### 4. DB 스키마는 `IF NOT EXISTS` + `ADD COLUMN IF NOT EXISTS`

기능 추가 시 기존 데이터를 건드리지 않는 안전한 마이그레이션.

### 5. `.single()` 대신 `.limit(1)` (supabase-py)

`.single()` 은 결과가 0건이면 예외 발생.  
`.limit(1)` + 빈 리스트 체크가 안전한 패턴.
